# ADR-0031: Generated document owner deletion

## Status

Accepted

## Date

2026-07-14

## Context

Generated documents are tenant-visible work products, but the history screen did not let a creator remove a document generated by mistake. The normal obra document delete flow is recoverable and keeps file metadata for restoration; deleting a generated-document history record also has to remove its PDF, upload tracking, and generated extraction rows so those representations do not drift apart.

## Decision

Allow an authenticated tenant member to permanently delete a `generated_documents` row only when `generated_by = auth.uid()`. This applies to every generated-document status, including approved documents, because ownership is the requested authorization boundary.

The generated-document API verifies the active tenant and ownership before using the service-role client for scoped artifact cleanup. It removes the stored PDF, `obra_document_uploads` tracking row, OCR processing rows, and `obra_tabla_rows` whose `data.__docPath` points to that PDF. Any existing file-trash history entry for the same path is marked purged. The API then deletes the generated-document row through owner-scoped RLS. Cascading generated-document events are deleted with the row. The source `generated_document_drafts` row is retained.

The history UI exposes the action only when the API reports `canDelete`, asks for confirmation, and states that deletion cannot be undone.

## Consequences

- Tenant members cannot delete generated documents created by another user, even when they can edit or review them.
- Deletion from generated history is a permanent cleanup operation and is distinct from the recoverable obra file-trash flow.
- Approved documents can be deleted by their creator; teams that need immutable approved-document retention require a separate retention policy.
- Storage and derived extraction cleanup is idempotent so a failed request can be retried while the generated-document row still exists.
- The database policy remains authoritative even if a client bypasses the history UI.

## Alternatives considered

Send the PDF to the existing obra trash while deleting the generated-document row. Rejected because restoring the file would not restore its generated-document metadata, events, or history entry.

Allow reviewers or tenant admins to delete any generated document. Rejected because the requested rule is creator ownership, while tenant-wide editing and review are separate capabilities.

Block deletion after approval. Rejected because it adds a lifecycle restriction not present in the requested ownership rule; immutable retention should be designed explicitly rather than inferred.
